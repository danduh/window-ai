## 1. Proofreader API

### Status

| Stage | Chrome versions | Access |
|-------|----------------|--------|
| EPP / local experimentation | 139 Canary+ | EPP flag only |
| Origin trial | 141–145 | OT token required |
| Post-145 | **unclear** — no "stable" announcement found as of this research | EPP or re-registration needed |

The origin trial ran through Chrome 145. As of Chrome 148/149 there is no confirmed stable shipping announcement for the Proofreader API (unlike the Prompt API which shipped stable in Chrome 148). The API is listed on the built-in AI docs page alongside Writer/Rewriter as a "Developer Trial" API and is under active development. Treat it as **EPP + OT-token-required** for v1.2 — check `developer.chrome.com/blog/proofreader-api-ot` for a live trial registration.

### Required flags (for localhost without OT token)

```
chrome://flags/#optimization-guide-on-device-model  →  Enabled
chrome://flags/#prompt-api-for-gemini-nano-multimodal-input  →  Enabled
chrome://flags/#proofreader-api-for-gemini-nano  →  Enabled
```

Relaunch Chrome after enabling. Verify model state at `chrome://on-device-internals`.

### Complete TypeScript interface (from W3C draft spec + WICG explainer)

```typescript
// Global: window.Proofreader

interface Proofreader {
  static create(options?: ProofreaderCreateOptions): Promise<Proofreader>;
  static availability(options?: ProofreaderCreateCoreOptions): Promise<AIAvailability>;

  proofread(input: string, options?: ProofreaderProofreadOptions): Promise<ProofreadResult>;
  // proofreadStreaming exists in the spec but NOTE: the Chrome 141-145 OT
  // implementation did NOT support it — only proofread() worked reliably.
  // proofreadStreaming(input: string, options?: ProofreaderProofreadOptions): ReadableStream<ProofreadResult>;

  readonly includeCorrectionTypes: boolean;
  readonly includeCorrectionExplanations: boolean;
  readonly expectedInputLanguages: ReadonlyArray<string> | null;
  readonly correctionExplanationLanguage: string | null;

  destroy(): void;
}

interface ProofreaderCreateCoreOptions {
  includeCorrectionTypes?: boolean;         // default: false
  includeCorrectionExplanations?: boolean;  // default: false
  correctionExplanationLanguage?: string;   // BCP-47 tag e.g. "en"
  expectedInputLanguages?: string[];        // BCP-47 tags e.g. ["en"]
}

interface ProofreaderCreateOptions extends ProofreaderCreateCoreOptions {
  signal?: AbortSignal;
  monitor?: (monitor: AICreateMonitor) => void;  // downloadprogress events
}

interface ProofreaderProofreadOptions {
  signal?: AbortSignal;
}

interface ProofreadResult {
  correctedInput: string;
  corrections: ProofreadCorrection[];  // ordered by ascending startIndex
}

interface ProofreadCorrection {
  startIndex: number;   // byte offset in original input string
  endIndex: number;     // exclusive
  correction: string;   // the replacement text
  types?: CorrectionType[];     // only present if includeCorrectionTypes === true
  explanation?: string;         // only present if includeCorrectionExplanations === true
}

type CorrectionType =
  | "spelling"
  | "punctuation"
  | "capitalization"
  | "preposition"
  | "missing-words"
  | "grammar";

// NOTE: spec has 6 values; Chrome stable impl in OT period returned only
// "spelling" and "grammar" reliably. preposition/missing-words are rare.

type AIAvailability = "available" | "downloading" | "downloadable" | "unavailable";
```

### Canonical usage pattern

```typescript
// 1. Check availability before showing UI
const avail = await Proofreader.availability({ expectedInputLanguages: ["en"] });
if (avail === "unavailable") {
  // hide proofreader UI entirely
  return;
}

// 2. Create — triggers LoRA download if needed (shows progress in monitor)
const proofreader = await Proofreader.create({
  includeCorrectionTypes: true,
  includeCorrectionExplanations: true,
  correctionExplanationLanguage: "en",
  expectedInputLanguages: ["en"],
  monitor(m) {
    m.addEventListener("downloadprogress", (e: DownloadProgressEvent) => {
      console.log(`LoRA download: ${e.loaded * 100}%`);
    });
  },
});

// 3. Proofread
const result: ProofreadResult = await proofreader.proofread(userText);

// result.correctedInput  → full corrected string
// result.corrections     → [{startIndex, endIndex, correction, types, explanation}, ...]

// Apply corrections (reverse-iterate so indices stay valid):
let annotated = userText;
for (const c of [...result.corrections].reverse()) {
  annotated =
    annotated.slice(0, c.startIndex) +
    `<mark data-type="${c.types?.[0]}">${c.correction}</mark>` +
    annotated.slice(c.endIndex);
}

// 4. Destroy when done
proofreader.destroy();
```

### Output characteristics

- **No confidence scores** — the spec and all Chrome implementations omit confidence/probability fields.
- **Corrections are NOT streaming** in the Chrome OT implementation — the full result arrives at once. The spec defines `proofreadStreaming()` but it was not functional during Chrome 141–145.
- **Corrections ordered ascending** by `startIndex`.
- **Chrome Stable vs Canary bug (observed in OT):** Chrome Stable occasionally returned only `correctedInput` with an empty `corrections` array. Canary returned both correctly. Pin to Canary for v1.2 demo.
- `explanation` fields are generative text from Gemini Nano — quality varies. They may be longer than the original error.

### Language support

From Chrome 149 Gemini Nano supports: **English, Spanish, Japanese, German, French** for input/output.  
Before Chrome 149: only English reliably supported.  
`expectedInputLanguages` is advisory — pass it always so the model can route correctly.

### Model download behavior

The Proofreader API depends on two download layers:

| Layer | Size | Download trigger |
|-------|------|-----------------|
| Gemini Nano base (`weights.bin`) | ~4 GB | Any built-in AI API `create()` call |
| Proofreader LoRA adapter | Small (minutes to download; exact MB unknown) | `Proofreader.create()` only |

LoRA is stored per-profile, separate from `OptGuideOnDeviceModel/`. It is downloaded alongside the base model on first `Proofreader.create()` and updated daily.

If Gemini Nano is already installed (user has used Summarizer/Prompt API), only the LoRA download is needed — much faster.

**Known issue:** LoRA download can stall at 0%. Workaround: verify base model is complete at `chrome://on-device-internals`, then retry `create()`.

**Storage purge:** Chrome deletes the base model if free disk drops below a threshold. After deletion, the model does **not** re-download automatically — `availability()` returns `"downloadable"`, not `"available"`.

### Hardware requirements

| Resource | Minimum |
|----------|---------|
| Free storage | 22 GB on Chrome profile volume |
| GPU VRAM | > 4 GB |
| OR: RAM | ≥ 16 GB + 4 CPU cores (CPU inference path) |
| OS | Windows 10/11, macOS 13+, Linux, ChromeOS Chromebook Plus |
| Mobile | Not supported |

## 2. LanguageModel Multimodal — Image Input

### Status

**Shipped stable in Chrome 148** (May 5, 2026). No flag required on Chrome 148+.

For Chrome 146/147 Canary (pre-stable), flags required:
```
chrome://flags/#optimization-guide-on-device-model  →  Enabled
chrome://flags/#prompt-api-for-gemini-nano           →  Enabled (or "Enabled multilingual")
chrome://flags/#prompt-api-for-gemini-nano-multimodal-input  →  Enabled
```

### Session creation

```typescript
const session = await LanguageModel.create({
  expectedInputs: [
    { type: "text", languages: ["en"] },
    { type: "image" },
    // { type: "audio" }  // only if audio needed; requires GPU
  ],
  expectedOutputs: [{ type: "text", languages: ["en"] }],
  monitor(m) {
    m.addEventListener("downloadprogress", (e) => { /* ... */ });
  },
});
```

`expectedInputs` must declare all modalities at creation time. Session creation:
- Triggers any additional model component downloads
- Fails with `NotSupportedError` if the device cannot support those modalities
- Returns synchronously if model already downloaded

### Supported image input types

Pass any of these directly as `value` in a content part:

```typescript
type ImageInputValue =
  | Blob           // ← most universal; works for File, fetch response blobs
  | HTMLImageElement
  | HTMLCanvasElement
  | OffscreenCanvas
  | ImageBitmap    // ← from createImageBitmap(file) or grabFrame()
  | ImageData
  | VideoFrame     // ← ideal for webcam (see below)
  | HTMLVideoElement  // snapshots current frame
  | SVGImageElement;

// BufferSource (ArrayBuffer / typed array) also accepted for raw image bytes
```

**Cross-origin note:** Images from cross-origin sources without `Access-Control-Allow-Origin` headers will be rejected with `SecurityError`. Canvas elements tainted by cross-origin draw calls also fail.

**Animated images:** Only the first frame is used.

### Prompt structure

```typescript
const response = await session.prompt([
  {
    role: "user",
    content: [
      { type: "text", value: "What do you see in this image?" },
      { type: "image", value: imageBlob },          // Blob from fetch or <input>
    ],
  },
]);

// OR streaming:
const stream = session.promptStreaming([
  {
    role: "user",
    content: [
      { type: "text", value: "Describe this frame:" },
      { type: "image", value: videoFrame },
    ],
  },
]);
for await (const chunk of stream) {
  // incremental text output
}
```

### Pre-processing with append()

For latency-sensitive flows, pre-submit the image before asking the question:

```typescript
// Pre-process image while user is typing their question
await session.append([{
  role: "user",
  content: [{ type: "image", value: capturedFrame }],
}]);

// Later, prompt with just the text — model already processed the image
const answer = await session.prompt("What changes since last frame?");
```

`append()` fulfills once the image is validated and processed. Use this for the webcam pipeline.

### Image format/dimension constraints

**Official documentation does not specify hard pixel limits or format whitelists.** Inference uses MIME sniffing (WHATWG rules). Unsupported formats throw `EncodingError`.

Practical constraints from community testing:
- JPEG, PNG, WebP work reliably via Blob
- Very high resolution images are accepted but dramatically increase processing time and token cost
- No documented max-dimension enforcement — but Gemini Nano internally downsamples for inference

### Memory and performance — webcam continuous capture

This is the critical concern for the webcam-live feature.

**Token cost of images:** A single high-resolution image can require 2,000+ tokens in a vision-language model, consuming transient VRAM above the base model allocation. Gemini Nano is a compact 2B/4B parameter model so absolute numbers are lower than server-side VLMs, but the pattern holds.

**Practical recommendations for continuous capture (every N seconds):**

```
Capture rate: 2–5 second intervals minimum
              (1 second intervals will cause inference backpressure)

Resolution:   Downsample to 512×512 or 640×480 before passing to model
              Use createImageBitmap(videoFrame, { resizeWidth: 512, resizeHeight: 512 })
              or draw to OffscreenCanvas at target resolution

Frame source: Use ImageCapture.grabFrame() → VideoFrame (lightweight)
              NOT takePhoto() (interrupts MediaStream, higher overhead)

Session:      Reuse a single session across captures
              Do NOT create a new session per frame

Backpressure: Gate next capture on previous prompt resolving
              (or use a simple queue with max depth 1)
```

**GPU requirement for image input:** Not explicitly stated in docs for image (unlike audio which explicitly requires GPU). CPU inference path may work for images on qualifying hardware (16 GB RAM, 4 cores), but performance will be much slower.

**Model size (base):** As of May 2026 Chrome ships ~4 GB Gemini Nano (`weights.bin`). Chrome selects between a 2B-parameter and 4B-parameter variant based on detected GPU performance class.

### Availability check pattern

```typescript
// Check before creating session
const avail = await LanguageModel.availability({
  expectedInputs: [{ type: "image" }],
});
// Returns: "available" | "downloading" | "downloadable" | "unavailable"

if (avail === "unavailable") {
  showFallbackUI();
} else if (avail === "downloadable") {
  showDownloadPrompt(); // 4 GB download required
}
```

### Controversy note (informational)

Chrome shipped Prompt API in Chrome 148 over objections from Mozilla, WebKit, W3C TAG, and Microsoft Edge (Edge disabled it). This does not affect functionality in Chrome but means the API is Chrome-specific for the foreseeable future — no cross-browser story. The `window-ai` project already assumes Chrome-only, so this is not a blocker.

## 3. Audio Input to Prompt API (v1.3 record)

**Status: Shipped in Chrome 148 stable**, alongside image input.

```typescript
const session = await LanguageModel.create({
  expectedInputs: [{ type: "audio" }],
  expectedOutputs: [{ type: "text" }],
});

const audioBuffer: AudioBuffer = await captureMicrophoneInput({ seconds: 10 });
const response = await session.prompt([{
  role: "user",
  content: [
    { type: "text", value: "Transcribe this:" },
    { type: "audio", value: audioBuffer },
  ],
}]);
```

**Accepted audio types:** `AudioBuffer`, `Blob`, `ArrayBuffer` / typed arrays (`BufferSource`).  
**Under investigation (not yet accepted):** `HTMLAudioElement`, `AudioData`, `MediaStream`.

**Critical constraint:** Audio input **requires a dedicated GPU**. Systems with only integrated graphics (e.g. Intel iGPU on Windows) return `NotAllowedError: Model capability is not available` even with all flags enabled.

**Practical fallback:** Web Speech API (speech-to-text) → Prompt API (text) is the reliable path for CPU-only hardware.

**Flags (Chrome 146/147 Canary, pre-148):**
```
chrome://flags/#prompt-api-for-gemini-nano-multimodal-input  →  Enabled
```
Same flag as image input.

**Note for v1.3 planning:** Audio input is technically available in Chrome 148 stable but has a hard GPU requirement that will silently fail on many demo machines. Design the feature with Web Speech API as primary path and native audio as progressive enhancement.

## 4. Open Questions for the Planner

1. **Proofreader OT post-145:** The origin trial officially ended at Chrome 145. As of Chrome 148/149 there is no confirmed stable shipping or new OT registration. Before building the Proofreader demo, confirm whether a new OT registration is active at `developer.chrome.com/origintrials/`. If not, the demo requires Chrome with the `#proofreader-api-for-gemini-nano` flag — viable for a Canary-gated showcase but not for general web users.

2. **Proofreader CorrectionType completeness:** The spec lists 6 types but Chrome 141–145 OT reliably returned only `"spelling"` and `"grammar"`. Test whether `"preposition"` and `"missing-words"` are returned in current Canary before building UI that handles all 6.

3. **Image dimension soft limit:** No official doc specifies max pixel dimensions. Community experience suggests large images (>1 MP) slow inference significantly. Run empirical tests with `createImageBitmap(blob, {resizeWidth: 512})` vs native resolution to find the acceptable latency/quality tradeoff for the webcam-live demo.

4. **Webcam-live memory on 8 GB VRAM machines:** Continuous 5-second capture at 512×512 with session reuse is expected to be viable on qualifying hardware. The 4 GB base model + image token overhead should stay within budget on 8 GB VRAM. Confirm empirically before publishing.

5. **Proofreader proofreadStreaming():** The spec defines it; Chrome 141–145 did not implement it. Check current Canary (146+) to see if streaming was added. If available, use it for responsive UX on longer documents.
