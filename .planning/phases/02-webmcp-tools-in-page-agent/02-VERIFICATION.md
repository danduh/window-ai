- **WR-01 (Concurrency race in mutators):** `executeScaleRecipe`/`executeSwapIngredient`/`executeAddIngredient`/`executeRemoveIngredient` do unsynchronized read-modify-write against IDB. The system prompt actively encourages concurrent tool calls. Concurrent mutators on the same recipe can clobber each other (one snapshot wins). Not a blocker — the canonical demo phrase invokes scaleRecipe + swapIngredient which produce different ingredient lists; AGENT-03 will likely succeed the first time. But UAT-04 step 8 fail-mode #2 ("Only one of the two tools fires") and stress repeats may surface this. Recommended fix in 02-REVIEW.md: wrap read+write in a single IDB readwrite tx.
- **WR-02 (liveToolName clobber on overlap):** Both onToolEvent dispatchers (page-level + drawer-side) track at most one in-flight tool; concurrent A/B calls produce a wrong "live = null" mid-flight when the first finishes. Not a blocker for the 2-min demo (the row highlight is decorative) but causes UAT-03 step 6 to be observed less reliably.
- **WR-03 (no AbortController for in-flight session.prompt):** `await session.prompt(text)` is not aborted on unmount; tool callbacks may run on unmounted state. Not a blocker for normal use; surfaces as React unmounted-setState warnings during fast navigation.
- **WR-04 (toolEvents map grows unbounded):** Indicator log accumulates dozens of rows over a long session; memory leak + UX clutter. Not a blocker for a 2-min demo.
- **WR-05 (executeSelectRecipe missing id type validation):** Defensive type-narrowing on `id` field. Inconsistent with sibling handlers but only matters when the model emits non-string ids (low probability). Not a blocker.

These warnings are tracked in 02-REVIEW.md and recommended for follow-up after Phase 3 ships. They do not gate the phase 2 acceptance.
