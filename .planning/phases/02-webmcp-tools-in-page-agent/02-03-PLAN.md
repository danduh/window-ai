**Change 3 — Add the tool-registration mount-effect** (insert AFTER the existing seed/load effect at line 80, BEFORE the `tabs` useMemo at line 85). This is the core MCP-02 wiring:

```tsx
// Tool registration mount-effect. Single AbortController per mount;
// controller.abort() in cleanup unregisters every tool atomically (per W3C
// WebMCP spec — there is no separate `unregisterTool` method).
// See 02-RESEARCH.md §Pattern 1 + §Pitfall 1 (empty deps) + §Pitfall 2 (controller inside effect).
useEffect(() => {
  if (typeof navigator === 'undefined' || !navigator.modelContext) {
    setRegistration({ status: 'unavailable', count: 0 });
    return;
  }
  const controller = new AbortController();
  // Wrap every tool with the SAME event dispatcher used by AgentDrawer's session,
  // so external-agent calls (Tool Inspector) also fire the indicator UI.
  const onToolEvent = (e: ToolCallEvent): void => {
    if (e.kind === 'pending') {
      setLiveToolName(e.toolName);
    } else {
      setLiveToolName(null);
    }
  };
  const wrapped = wrapToolsWithEvents(RECIPE_TOOLS, onToolEvent);
  const registered: string[] = [];
  try {
    for (const tool of wrapped) {
      navigator.modelContext.registerTool(tool, { signal: controller.signal });
      registered.push(tool.name);
    }
    setRegistration({ status: 'success', count: registered.length });
  } catch (err) {
    const message = err instanceof Error ? err.message : 'Unknown error';
    // eslint-disable-next-line no-console
    console.error('[RecipeWorkbench] Tool registration failed:', message);
    setRegistration({
      status: registered.length > 0 ? 'partial' : 'error',
      count: registered.length,
    });
  }
  return () => controller.abort();
  // eslint-disable-next-line react-hooks/exhaustive-deps
}, []);
```

**Critical constraints (RESEARCH-verified):**
- Empty deps `[]` — RECIPE_TOOLS is module-static; re-running this effect would throw `InvalidStateError` because the same tool name can't be registered twice on the same `ModelContext` (Pitfall #1).
- `new AbortController()` MUST live INSIDE the effect — hoisting it would re-use an already-aborted controller after StrictMode double-mount (Pitfall #2).
- `if (!navigator.modelContext)` short-circuit MUST set status to `'unavailable'` so the pill knows to hide and the ToolListPanel knows to show the unavailable copy.
- Catch handler MUST distinguish `partial` (some succeeded) from `error` (none succeeded) using the `registered.length` check.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| External agent (Tool Inspector / future MCP client) → registered tool via `navigator.modelContext` | Untrusted JSON args cross from outside the page into our handler chain (via the same `wrapToolsWithEvents` wrapper feeding the same handler bodies as the in-page agent). Same threat model as the in-page agent path. |
| `controller.signal` lifetime → registered-tools map | A leaked or re-used controller could result in tools that fail to deregister (orphaned in the page's modelContext map). |
| AbortController → React StrictMode double-mount | StrictMode invokes effects twice in dev; if the controller is hoisted, the second effect sees an already-aborted signal and silently fails to register. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03-01 | Tampering | External-agent tool invocation via Tool Inspector / future MCP client | mitigate | Same handler chain as the in-page agent — input validation in `recipeToolHandlers.ts` (Plan 02-01) is the single guard. Wrapped via `wrapToolsWithEvents` so all errors land in `{error: "..."}` instead of throwing across the boundary. |
| T-02-03-02 | Spoofing | Tool name collision after SPA navigation | mitigate | `controller.abort()` runs on unmount (React's component cleanup); SPA route change unmounts the page, releasing all 8 tool names. Re-mount creates a fresh controller. RESEARCH §Pitfall #8 verified. |
| T-02-03-03 | Tampering | Re-registration on every render | mitigate | `useEffect` empty deps + module-static `RECIPE_TOOLS` ensures a single registration per mount lifecycle (RESEARCH §Pitfall #1). |
| T-02-03-04 | Tampering | StrictMode double-invoke leaving an orphaned controller | mitigate | `new AbortController()` lives INSIDE the effect; the abort + re-create on the second StrictMode invocation produces a fresh controller (RESEARCH §Pitfall #2). |
| T-02-03-05 | Information Disclosure | Console error logging during registration failure | mitigate | Log only `err.message` (string), not the full error object (matches Phase 1 convention). |
| T-02-03-06 | Denial of Service | Adversarial recipe data via Tool Inspector executions | accept | Local-only demo, no multi-tenant surface; IndexedDB write is bounded by browser quota. |
| T-02-03-07 | Elevation of Privilege | navigator.modelContext.registerTool throwing in Secure-Context-mismatched environment | mitigate | try/catch around the registerTool loop; partial-vs-error branch sets the pill state; UAT-01 fail-mode A captures it for the user. RESEARCH §Pitfall #9. |

ASVS L1 alignment: input validation at the trust boundary (T-02-03-01 — handled in 02-01), lifecycle hygiene (T-02-03-02..04), error-logging discipline (T-02-03-05).
