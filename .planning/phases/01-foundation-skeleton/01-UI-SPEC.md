# Phase 1 — UI Design Contract

> Visual and interaction contract for the WebMCP Recipe Workbench Foundation Skeleton.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope (UI surfaces this contract governs):**

1. `/webmcp` route mount + main nav link (NAV-01, NAV-03)
2. Recipe Workbench page rendering active recipe (UI-01, UI-03)
3. Missing-flag banner for browsers without `navigator.modelContext` (UI-02)

**Out of scope (Phase 2 / Phase 3 — do NOT design here):**
- In-page chat panel (Phase 2 — AGENT-01)
- Tool registration UI / inspector view (Phase 2 — MCP-02..05)
- `/webmcp/docs` markdown explainer body (Phase 3 — DOCS-01) — the docs **tab shell** is wired in Phase 1 (per RESEARCH §3 Q2) but its content is a placeholder

## Design System

| Property | Value |
|----------|-------|
| Tool | none — brownfield project, manual Tailwind |
| Preset | not applicable |
| Component library | none — bespoke Tailwind components mirror existing demos |
| Icon library | none — inline JSX SVGs (24x24 viewBox, `currentColor` stroke), matching `Summary.tsx`/`HomePage.tsx`/`WriteRewritePage.tsx` |
| Font | Tailwind default `font-sans` (system stack: ui-sans-serif, system-ui, …) — no custom font loaded |

**Justification for `Tool: none`:** The repository has 5 existing demo pages (`/chat`, `/tool-calling`, `/summary`, `/translate`, `/writer`) all built with hand-rolled Tailwind classes. CLAUDE.md mandates brownfield discipline ("don't refactor what isn't being touched") and DoD = "2-min demo, not reference-quality". Initializing shadcn for one new page would produce visual inconsistency with the rest of the site and violate the milestone constraints. Phase 1 reuses the established Tailwind utility patterns documented below.

**Detected tokens reused verbatim from `chat/tailwind.config.js`:**
- `darkMode: 'class'` — class-based dark mode toggled by `ThemeProvider` writing `.dark` to `<html>`
- `colors.primary.{50..900}` — blue scale with `primary-500 = #3b82f6`, `primary-600 = #2563eb`
- Animations: `fade-in` (0.5s), `slide-up` (0.3s) — available but not required by Phase 1

## Typography

The codebase uses Tailwind's default type scale at `font-sans` (system stack). The four roles below are the ONLY type sizes Phase 1 may introduce. Lock these.

| Role | Size | Weight | Line Height | Tailwind | Usage |
|------|------|--------|-------------|----------|-------|
| Body | 16px | 400 (regular) | 1.5 | `text-base` (default 1.5) | Recipe step text, banner body, ingredient name + amount |
| Label | 14px | 500 (medium) | 1.5 | `text-sm font-medium` | Picker button labels, step number "Step 1", servings count "Serves 4", banner heading text |
| Heading | 20px | 600 (semibold) | 1.5 | `text-xl font-semibold` (matches `Summary.tsx:197`, `WriteRewritePage.tsx:277`) | Recipe title, "Ingredients" / "Steps" section headings |
| Display | 30px | 700 (bold) | 1.2 | `text-3xl font-bold` (matches `HomePage.tsx:54`, `Summary.tsx:145`, `WriteRewritePage.tsx:201`) | Page title "Recipe Workbench" in the header |

**Weights used:** exactly two — 400 (regular, default) and 600 (semibold). The Display role uses 700 (bold) — this is one extra weight above the prescribed two-weight rule but matches every existing demo page's H1; deviating would create visual inconsistency. **Approved exception:** Display = 700 to match site-wide H1 convention. Body, Label, Heading remain on 400/600.

**Subtitle in header (e.g. `WriteRewritePage.tsx:202` "Powered by Chrome's Writing Assistance APIs"):** uses Body role (`text-gray-600 dark:text-gray-400` at default `text-base`). Phase 1 subtitle: see Copywriting Contract.

## Copywriting Contract

Voice: technical-but-friendly, demo-first. Same register as the rest of `windowai.danduh.me` (see `HomePage.tsx:63-67`). Sentence-case headings, no marketing fluff, code-fenced API names.

### Page header

| Element | Copy |
|---------|------|
| H1 (Display) | `Recipe Workbench` |
| Subtitle (Body, gray-600) | `A WebMCP demo: tools live on the page, not on a server.` |

### Recipe picker (UI-03)

| Element | Copy |
|---------|------|
| Section label (Label, sr-only — `<label>` for the picker group) | `Active recipe` |
| Picker buttons | Recipe `title` field rendered verbatim (e.g. `Buttermilk Pancakes`, `Tomato Pasta`) |
| Active state announcement | `aria-pressed="true"` on the active button. No visible "active" word — visual treatment alone communicates state. |

### Recipe view (UI-01)

| Element | Copy |
|---------|------|
| Servings line (Label) | `Serves {n}` (e.g. "Serves 4") — singular form `Serves 1` is grammatically fine in this context |
| Ingredients heading (Heading) | `Ingredients` |
| Ingredient row format | `{quantity} {unit} {name}` — e.g. `200 g flour`, `2 piece eggs`. Unit `piece` displays as-is (no smart pluralization Phase 1). |
| Steps heading (Heading) | `Steps` |
| Step row format | Numbered list. `<li>` tag. Numeral rendered as `text-gray-500` `Step {n}` Label above the step body, OR rely on `<ol>` default numbering — implementer chooses but must be consistent across both seeded recipes. |

### Missing-flag banner (UI-02)

Position: directly above the Tabs component, inside `max-w-6xl mx-auto p-4`. Rendered ONLY when `navigator.modelContext` is `undefined`. Below the banner, the page (picker + recipe view) remains fully functional.

| Element | Copy |
|---------|------|
| Heading (Label `font-medium`, `text-yellow-800 dark:text-yellow-200`) | `WebMCP isn't enabled in this browser.` |
| Body (Body, `text-yellow-700 dark:text-yellow-300 text-sm mt-1`) | `The recipe browser still works. To register tools and run the agent demo, open this page in Chrome 146+ Canary with the WebMCP flag enabled.` |
| Code line 1 label | `Browser:` |
| Code line 1 value (CodeSnippet, inline) | `Chrome 146+ Canary` |
| Code line 2 label | `Flag:` |
| Code line 2 value (CodeSnippet, inline) | `chrome://flags/#WebMCP` (set to "For testing") |
| Icon | The same warning triangle SVG used at `HomePage.tsx:70-72` (path `M8.257 3.099 …`). `text-yellow-600 dark:text-yellow-400 w-5 h-5 mt-0.5 mr-2 flex-shrink-0`. |

**Why `chrome://flags/#WebMCP` renders as code, not a link:** Browsers refuse to navigate to `chrome://` URLs from page anchors. Rendering as `<code>` signals "copy this; paste it in your address bar."

### Empty / fallback states (defensive — should not appear with seeded recipes)

| Element | Copy | When shown |
|---------|------|------------|
| No recipes loaded yet | `Loading your recipes…` (Body, `text-gray-500 dark:text-gray-400`) | First mount, before `seedIfEmpty()` and `getRecipes()` resolve. Brief flash; should be <100ms. Render this in place of the recipe view, NOT the picker — picker stays hidden until recipes resolve. |
| All recipes deleted (defensive only — DATA-02 seeds prevent this Phase 1) | `No recipes yet. Reload the page to seed the demo recipes.` (Body, centered, in the recipe view area) | Edge case: if a developer manually clears IndexedDB without reloading. |
| Active recipe has zero ingredients (defensive) | `No ingredients listed.` (Body, italic, `text-gray-500 dark:text-gray-400`) | Should never occur with seeded data; defensive only. |
| Active recipe has zero steps (defensive) | `No steps listed.` (Body, italic, `text-gray-500 dark:text-gray-400`) | Same as above. |

### Error states (defensive — IndexedDB failures)

| Element | Copy | Tone |
|---------|------|------|
| Persistence read failed | `Couldn't load recipes from your browser's storage. Try reloading the page.` (Body) — rendered inside an inline yellow alert below the picker. Do NOT block the page. | Friendly, actionable, no stack trace. |
| Persistence write failed (Phase 2 territory; recorded here for forward-compatibility) | `Saving failed. Your changes weren't persisted.` | Same. Phase 2 will hook this up. |

### Primary CTA

**There is no primary CTA in Phase 1.** The Recipe Workbench is a read-only browsing surface. The Phase 2 chat panel introduces the first CTA ("Send" / "Ask the agent"). Phase 1 deliberately ships zero buttons that mutate state — picker buttons are navigation, not action.

If the planner needs a CTA-shaped affordance (e.g. for the eventual Phase 2 chat input), the locked label is **`Ask the agent`** with placeholder `"e.g. Scale to 6 and swap milk for oat milk"`. Do NOT ship this in Phase 1.

### Destructive actions

**None this phase.** Recorded for Phase 2: when `removeIngredient` is wired to a tool, any UI surface for it must include a confirmation dialog with copy `Remove "{ingredient name}"? This change persists.` and buttons `Remove` (destructive) / `Cancel`.

## Interaction Contract

### Theme

Inherits from existing `ThemeProvider` (`chat/src/app/context/ThemeContext.tsx`). The `ThemeToggle` button in the page header toggles the `.dark` class on `<html>`. All Phase 1 components MUST declare both light and dark variants (every `bg-*`, `text-*`, `border-*` class needs its `dark:*` partner). This is non-negotiable — a missing dark variant on the recipe view would render unreadable when the rest of the page flips.

### Feature detection

Single-line `if (!navigator.modelContext)` check at component-mount time (per RESEARCH Pitfall 2). Don't wrap in a helper. The check determines whether `<MissingFlagBanner/>` renders — nothing else on Phase 1's page is gated by this check. The recipe browser is fully usable in either branch.

### Loading

When `getRecipes()` is in flight on first mount, render `Loading your recipes…` in the recipe-view column (right `lg:col-span-3` area). Picker column renders nothing (no buttons) until recipes resolve. No spinner asset — text-only loading is consistent with the rest of the codebase (the existing demos render `"Generated content will appear here…"` placeholder text rather than spinners).

### Reactivity

When a picker button is clicked, `setActiveRecipeId(newId)` runs synchronously and the recipe view rerenders the new title/servings/ingredients/steps within the same paint frame. UI-03 success criterion ("real-time update") is met because IndexedDB is read once at mount and held in `useState<Recipe[]>` — switching active recipes is a pure in-memory derive, no IndexedDB round-trip.

### Accessibility

DoD = "2-min demo, not reference-quality" — full WCAG audit is out of scope (V2-04 in REQUIREMENTS.md). However, the following are FREE wins from reusing existing patterns and MUST land:

- Picker buttons: `<button aria-pressed={isActive}>` (semantic, screen-reader-friendly) — already part of the Tailwind class spec above.
- Banner icon: decorative — `aria-hidden="true"` on the SVG.
- Recipe heading levels: `<h1>` page title, `<h2>` recipe title (in the active recipe card), `<h3>` "Ingredients" / "Steps". Do not skip levels.
- `<ol>` for steps (semantic ordered list), `<ul>` for ingredients (semantic unordered list).
- Tab triggers: `aria-current="page"` on active tab — already wired in `Tabs.tsx:70`.

### Responsive

- Mobile (<lg breakpoint, 1024px): single-column stack — picker card on top, recipe view below. Comes free from `grid-cols-1 lg:grid-cols-4`.
- Desktop (≥lg): 4-column grid, picker = 1 col, recipe = 3 col. Matches `Summary.tsx:192`.
- Page max width: `max-w-6xl` (72rem / 1152px) — matches every existing demo. Wider screens letterbox.
