<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM output (parsed JSON) → host tool dispatcher | Untrusted: the model can emit any `toolName`. Mitigated by lookup against `visibleTools` (visibility:['app'] filtered) and a name-not-found fallback that does not execute anything. |
| Tool result `_meta` field → next-turn prompt | Untrusted: `_meta` may contain `ui://` URIs or arbitrary keys. Mitigated by the interceptor that strips `_meta` before building `toolResultForModel` — only `result.content` is fed back. |
| Iframe (sandboxed, null origin) → `tools/call` over postMessage | Phase 5 bridge already enforces `event.source === outerIframe.contentWindow` origin guard. No Phase 6 change. |
| MealPlanStore.addToPlan → onCommitCallback → addMessage | The recipeId originates from the iframe carousel's `data-recipe-id` attribute (sandboxed). Treated as untrusted: `getRecipe(recipeId)` returns undefined for unknown ids, falling back to `${recipeId}` in the confirmation string (no XSS — message text is rendered through React, not innerHTML). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-06 | Information disclosure | `_meta.ui.resourceUri` leaks into `session.prompt()` argument; model emits it in `reply` | mitigate | The `_meta` interceptor in GenUIChatPanel strips `_meta` before building `toolResultForModel`. A `console.assert(!promptText.includes('ui://'))` BEFORE every `session.prompt()` call makes the invariant load-bearing at runtime in dev builds. Acceptance criterion: grep on source returns 0 occurrences of `ui://` inside any `promptText = ...` expression. |
| T-06-07 | Spoofing | LLM fabricates a `_meta.ui.resourceUri` for a tool that didn't actually produce one (model hallucination), tricking ChatBox into rendering a forged URI | accept | The URI lookup goes through `recipeCarouselRegistry.getRecipes(token)`. A token not present in the registry returns `undefined` and the helper returns null → ChatBox falls back to Markdown rendering. No iframe renders for forged URIs. |
| T-06-08 | Elevation | A tool with `visibility:['app']` annotation accidentally surfaces in the SYSTEM_PROMPT, exposing privileged operations to the model | mitigate | The `visibleTools` filter (`!t.annotations?.visibility?.includes('app')`) is applied at BOTH the SYSTEM_PROMPT construction site AND the tool-name lookup site inside the dispatch loop. Both filters must pass for a tool to be reachable from the chat. Verified by acceptance grep. |
| T-06-09 | Tampering | The model emits a forged `commitRecipeToPlan` call from chat (bypassing the visibility hide) | mitigate | The tool-name lookup against `visibleTools` returns undefined for `commitRecipeToPlan`. The dispatch loop's "Tool not registered" branch fires; nothing executes. The legitimate path remains the iframe `tools/call` over the Phase 5 bridge. |
| T-06-10 | Denial of service | An LLM stuck in a non-`done` loop runs the dispatch forever | mitigate | `MAX_TOOL_ITERATIONS = 5` caps the loop. After 5 turns the panel forces a fallback message. Mirrors AgentDrawer's `MAX_TOOL_CALLS = 10` safety pattern. |
| T-06-11 | Repudiation | The "Added X ✓" message is added even when `MealPlanStore.addToPlan` silently failed | accept | `setCommitListener`'s callback is invoked from inside `commitRecipeToPlan.execute` AFTER `await MealPlanStore.addToPlan(...)` resolves. If `addToPlan` throws, `wrapToolsWithEvents` swallows it and returns `{ error }`, so the callback is NOT invoked and the chat does NOT show the confirmation. Acceptable risk: the chat may show no feedback in this rare failure case. |
